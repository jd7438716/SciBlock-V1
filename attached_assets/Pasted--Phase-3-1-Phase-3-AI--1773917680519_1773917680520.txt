现在开始 Phase 3.1：实验报告可读性与结构化表达优化。

前提说明：
Phase 3 已经完成了实验报告内容质量优化，当前状态是：
- 报告已可持久化
- AI 只生成实验概述 / 结果分析 / 实验结论三个核心区块
- 后端有 evidenceLevel（high / medium / low）分级
- 缺失项 missingInfoHints 已进入生成链路
- 输出内容相比之前更保守、更可信，也更能回应 objective

所以这次不要再做大功能扩展，也不要重构底层架构。  
Phase 3.1 的目标只有一个：

把实验报告从“内容可用”进一步优化为“结构更清晰、阅读更顺畅、表达更像正式报告”的版本。

一、Phase 3.1 总目标

重点优化以下三个方面：

1. 报告阅读结构更清楚
当前 analysis / conclusion 仍然偏单段文字，阅读层次不够清晰。需要让用户更容易一眼看懂：
- 当前分析是什么
- 证据边界在哪里
- 当前能下什么程度的结论
- 还缺哪些信息

2. evidenceLevel 从“埋在正文里的说明”升级为“明确但克制的 UI 信号”
现在 evidenceLevel 只是写进 HTML 文本里，能用，但产品感较弱。需要变成轻量、清晰、不喧宾夺主的视觉提示。

3. missingInfoHints 从“附带建议文字”升级为“更像报告附注/补充建议”
让用户更直观看到“接下来该补什么”，而不是混在正文里读过去。

二、这次先做代码审查，再实施

请先基于当前真实代码审查以下几点，再动手：

1. 当前 renderer 是如何渲染 analysis / conclusion 的
2. 当前 evidenceLevel 是在哪里被插入到 HTML 的
3. 当前 missingInfoHints 是如何渲染的
4. 当前 report model / AI 返回结构，是否已经足够支撑更清晰的区块表达
5. 当前是否可以在不改大架构的前提下，做轻量结构优化

三、analysis / conclusion 的结构化表达优化

这次不要做很重的 schema 设计，但请把 analysis / conclusion 从“单段长文本”尽量优化为更有层次的表达。

目标是让它更接近这种阅读结构：

结果分析
- 当前分析
- 证据边界 / 局限说明

实验结论
- 阶段性结论
- 后续验证或补充建议

实现上你可以选择以下任一合理方式：

方案 A（优先推荐）：
让 AI 返回更细一点的 JSON 结构，例如：
- analysisMain
- analysisLimitations
- conclusionMain
- conclusionNextSteps

然后 renderer 把它们渲染成两个小段落或小列表。

方案 B（如果你认为改 AI 返回结构风险较大）：
保持 AI 仍返回 analysis / conclusion 单字段，但在后端或 renderer 中做轻量分段处理，比如：
- 优先拆成主判断 + 限制说明
- 或者按 evidenceLevel 决定固定展示骨架

要求：
- 不要把报告变成很碎的表单风
- 仍然要保持“正式实验报告”的文字感
- 但要明显比现在更易读

四、evidenceLevel 的 UI 化

请把 evidenceLevel 从纯正文提示，升级为轻量 UI 标签，但不要做复杂设计改版。

要求：

1. 标签要克制
建议展示文案类似：
- 信息充分
- 信息有限
- 信息不足

2. 标签位置合理
可选位置：
- 报告顶部（靠近标题或元信息）
- 或“结果分析”区块标题旁
- 或“结果分析/实验结论”前的轻量提示条

请选择一个最合适的位置，不要全放。

3. 风格要求
- 低调、学术、克制
- 不要做得像告警弹窗
- 不要颜色过重
- 不要抢占正文注意力

4. 保留正文中的保守表达
UI 标签不是替代内容本身。
正文仍然要保留“初步判断 / 信息不足 / 有待验证”的文字约束，只是不要再完全依赖一行正文提示承担全部语义。

五、missingInfoHints 的展示优化

当前已有 missingInfoHints，但请把它做得更像正式报告中的“补充说明 / 后续建议”。

要求：

1. 位置要固定且合理
建议放在：
- 结论区块末尾
- 或作为结论后的“后续建议”小节

2. 表达方式更清晰
不要只是斜体一行带过。
可以考虑：
- 简短列表
- 或“建议补充信息：”下的 2~4 条要点

3. 与 evidenceLevel 联动
- high：可以不展示，或弱展示
- medium：展示 1~3 条关键建议
- low：展示更明确的补充建议

4. 不要过度产品化
它仍然是报告的一部分，不要做成任务管理面板风格。

六、报告顶部的信息组织微调

在不大改布局的前提下，请顺手检查并优化报告顶部信息区的可读性：

1. 标题、实验类型、生成时间、source 的显示层级是否清楚
2. 如果 evidenceLevel 选择放顶部，是否会挤乱头部布局
3. “AI 生成 / AI 修改后 / 手动”等状态是否需要更清楚但克制地展示

要求：
- 不要新增过多标签
- 不要让顶部过于拥挤
- 只做必要优化

七、renderer 层要求

这次的优化尽量集中在 renderer / 轻量 model 结构层，不要把格式逻辑塞进 UI 组件里。

建议仍然遵守分层：
- AI 返回结构
- report model
- renderer 输出 HTML
- UI 只负责展示

不要把 analysis / conclusion 的分段规则硬写进 React 组件里。

八、这次不要做的事

本轮不要做：
- 报告版本历史
- PDF / Word 导出
- 局部区块重生成
- 差异对比视图
- 自动保存 typing
- 导师批注实验报告
- 复杂富文本模板系统
- 大规模 UI 改版
